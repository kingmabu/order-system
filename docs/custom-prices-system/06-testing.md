# 06. テスト項目・段階的展開手順

このファイルでは、開発完了から本番展開までの**段階的なテスト戦略**と各段階のチェックリストを定義します。

**実装に着手する前に、必ずこのファイル全体を読んでください。**

---

## 1. テスト戦略の全体像

```
[段階1] 開発環境での機能テスト（dry-run）
  ↓
[段階2] 開発環境での業務シナリオテスト
  ↓
[段階3] 本番環境への移植
  ↓
[段階4] 本番環境で自社（MY Inc.）テスト注文
  ↓
[段階5] 先行顧客1〜2社で実運用テスト（1週間程度）
  ↓
[段階6] Individual全15社へ展開
  ↓
[段階7] Group A・Standard顧客も含めた全顧客展開
  ↓
[段階8] Choco解約
```

各段階で問題があれば前段階に戻ります。**急がず、段階ごとに確実に確認**することが重要です。

---

## 2. 段階1：開発環境での機能テスト（dry-run）

### 目的
個別機能が単体で正しく動くかを確認します。

### 前提条件
- 開発用 Cost list / Item List / Client Information にコピーが完成している
- `feature/custom-prices` ブランチで実装が完了している
- `.env.development` が設定済み
- `QBO_MODE=dry-run` になっている

### 2-1. Cost list 機能テスト

開発用 Cost list で確認：

- [ ] メニュー「価格管理」が表示される
- [ ] 「個別価格を追加」フォームが起動する
- [ ] 顧客プルダウンに Individual分類の顧客のみ表示される
- [ ] 顧客プルダウンが空の場合、メッセージが表示される
- [ ] SKU検索ボックスでオートコンプリート動作（SKU/商品名どちらも）
- [ ] 商品選択時に参考情報（仕入価格・標準販売価格・粗利率）が表示される
- [ ] 量り売り商品で「販売単位：量り売り（ポンド単価）」と表示
- [ ] 定量売り商品で「販売単位：定量売り（箱単価）」と表示
- [ ] 個別価格を追加 → Custom Prices シートに記録される
- [ ] 同時に Custom Price Log に Action=Add のログが記録される
- [ ] Update Date に本日の日付が自動入力される
- [ ] 同じ Customer ID + SKU で重複追加 → 「変更画面に切替」ダイアログ表示
- [ ] 「変更画面に切替」を押すと変更モードで開く
- [ ] 価格を変更 → Custom Prices シートの該当行が更新される
- [ ] 変更ログに Price Before と Price After の両方が記録される
- [ ] 価格を削除 → Custom Prices シートから該当行が消える
- [ ] 削除ログに Action=Delete・Price Before が記録される
- [ ] 「Cost Reference を更新」で Cost Reference シートが正しく集計される
- [ ] Cost Reference が無い状態で「価格管理 → 追加」を押すと自動生成される
- [ ] 「他の顧客を追加」リンクで任意の顧客をIndividualに変更できる
- [ ] Sort & Clean を全タブに実行しても、新規シート3つは整列対象外
- [ ] Preferred Price 再構築でも、新規シート3つはスキャン対象外
- [ ] Vendor Formulas 「全タブに数式を適用」でも新規シート3つは対象外

### 2-2. 入力検証テスト

- [ ] 価格を空欄で保存 → 「価格を入力してください」エラー
- [ ] 価格に 0 を入力 → 「価格は0より大きい数値で入力してください」エラー
- [ ] 価格に -10 を入力 → 同上エラー
- [ ] 価格に "abc" を入力 → 同上エラー
- [ ] 価格に $1500 を入力 → 「価格が高額です。よろしいですか？」確認ダイアログ
- [ ] 仕入価格 $10 の商品に $8 を入力 → 「赤字売りになります」確認ダイアログ
- [ ] ダイアログでキャンセル → 保存されない
- [ ] ダイアログでOK → 保存される

### 2-3. order-system 機能テスト

ローカルで `node server.js` を起動して curl でテスト：

- [ ] Standard顧客の注文 → ベース価格（J列 or K列）が反映される
- [ ] Group A顧客の注文 → ベース価格 × 1.015 が反映される
- [ ] Individual顧客（Custom Price あり）→ Custom Price が反映される
- [ ] Individual顧客（Custom Price なし）→ ベース価格にフォールバック
- [ ] フォールバック時に warning メッセージが出る
- [ ] 量り売り商品 → J列（Price $/lb）の単価が使われる
- [ ] 定量売り商品 → K列（Unit Price）の単価が使われる
- [ ] dry-runモード → コンソールに `[DRY-RUN]` ログが出力される
- [ ] dry-runモード → 実際に QBO に送信されない
- [ ] dry-runモード → レスポンスに `dryRun: true` が含まれる
- [ ] エラー時にメール通知が `ordercfp@gmail.com` に届く
- [ ] order-system record（開発用）に処理状況が記録される
- [ ] Customer ID が見つからない → Standardで処理 + 警告メール
- [ ] SKU が見つからない → 該当行スキップ + エラーメール

### 2-4. リトライ動作テスト

- [ ] Sheets API を一時的に遮断（ネットワーク切断）→ 3回リトライ後にエラー
- [ ] リトライ間隔が指数バックオフ（1秒 → 3秒 → 9秒）になっている

### 2-5. パフォーマンステスト

- [ ] 5アイテムの注文 → 2秒以内に応答
- [ ] 10アイテムの注文 → 3秒以内に応答
- [ ] 20アイテムの注文 → 5秒以内に応答
- [ ] 連続10件の注文 → エラーなく処理

### 段階1の完了条件

上記すべての項目に ✅ が付いたら段階2へ。

## 3. 段階2：開発環境での業務シナリオテスト

### 目的
実際の業務に近いシナリオで、複数機能が連携して正しく動くかを確認します。

### 3-1. シナリオA：新規顧客の初期セットアップ

```
[手順]
1. 新規顧客「TEST RESTAURANT」を Client list に追加（Customer ID = 999）
2. Price Group を最初「Standard」に設定
3. server.js から TEST RESTAURANT で注文 → Standard価格で処理されることを確認
4. Price Group を「Group A」に変更
5. 同じ注文で +1.5% が適用されることを確認
6. Price Group を「Individual」に変更
7. Custom Prices に1件追加（例：B033 → $22.50）
8. 同じ注文で Custom Price が適用されることを確認
9. Custom Price を削除
10. 同じ注文で Standard にフォールバックすることを確認
```

- [ ] 全10ステップが期待通り動く

### 3-2. シナリオB：複数アイテム混在注文

```
[手順]
1. Individual顧客（例：011 BENI）が以下を注文：
   - B033 (Custom Price あり) × 2
   - C037 (Custom Price あり) × 3
   - B008 (Custom Price なし、定量売り) × 1
   - C001 (Custom Price なし、量り売り) × 5
2. server.js を実行
3. レスポンスを確認
```

- [ ] B033: Custom Price で計算されている
- [ ] C037: Custom Price で計算されている
- [ ] B008: K列（Unit Price）で計算されている
- [ ] C001: J列（Price $/lb）で計算されている
- [ ] フォールバック警告がレスポンスに含まれる（B008, C001）

### 3-3. シナリオC：価格変更後の即時反映

```
[手順]
1. Individual顧客の Custom Price を変更（例：$22.50 → $21.00）
2. server.js を再起動せずに、同じ注文を実行
3. 新価格が反映されることを確認
```

- [ ] キャッシュなしの実装どおり、価格変更が即反映される

### 3-4. シナリオD：エラーリカバリ

```
[手順]
1. server.js 実行中に、Sheets API への接続を一時的に遮断（Wi-Fi切るなど）
2. 注文を実行
3. リトライが3回行われ、最終的にエラー応答が返ることを確認
4. アラートメールが届くことを確認
5. 接続を復旧
6. 再度同じ注文を実行 → 正常応答
```

- [ ] エラー時に処理が止まる
- [ ] アラートメールが届く
- [ ] 復旧後は正常動作

### 段階2の完了条件

4つのシナリオすべてが期待通り動いたら段階3へ。

## 4. 段階3：本番環境への移植

### 4-1. 移植内容

| 対象 | 内容 |
|---|---|
| **本番 Cost list** | Apps Script 改修（既存ファイル4 + 新規ファイル3）|
| **本番 Client Information** | Client list に X列・Y列を追加 + 全顧客分類 |
| **本番 Cost list** | Custom Prices に CSV 投入（開発用と同じ内容）|
| **`main` ブランチ** | `feature/custom-prices` をマージ |
| **Render** | `.env.production` を設定し自動デプロイ |

### 4-2. 移植手順（詳細）

#### Step 1：本番 Cost list の Apps Script 改修

1. 本番 Cost list を開く
2. Apps Script エディタを開く
3. 既存4ファイルに除外リスト追加（開発用と同じ変更）
4. 新規3ファイルをコピー＆ペースト：
   - `cost_reference.gs`
   - `custom_prices_form.gs`（**`CLIENT_INFO_ID` と `ITEM_LIST_ID` を本番IDに変更**）
   - `custom_prices_form.html`
5. 保存
6. メニューバーに「価格管理」が表示されることを確認
7. 「Cost Reference を更新」を実行 → Cost Reference シートが生成されることを確認

#### Step 2：本番 Client list に列追加

1. 本番 Client Information を開く
2. Client list シートのX列・Y列に手動でヘッダー追加：
   - X1：`Price Group`
   - Y1：`Markup %`
3. X列に **データの入力規則（プルダウン）** を設定：
   - 候補：`Standard`, `Group A`, `Individual`
4. 全顧客の Price Group を分類：
   - Group A の10社 → `Group A` + Y列に `1.5`
   - Individual の15社 → `Individual`
   - 残り → `Standard`

#### Step 3：本番 Custom Prices に投入

1. 開発で動作確認済みの CSV を用意
2. 本番 Custom Prices シートにインポート（**段階1で確認したのと同じCSV**）
3. データ確認（行数・Customer ID・SKU・価格）

#### Step 4：order-system 本番デプロイ

```bash
git checkout main
git merge feature/custom-prices
git push origin main
```

→ Render が自動デプロイ。

#### Step 5：Render の環境変数設定

Render のダッシュボードで以下を設定：

```
NODE_ENV=production
COST_LIST_ID=1dC88enQnxjK8-GgxQhA6z4xiICUZ-ShFGnzcYySY73k
ITEM_LIST_ID=14dKo33uLpVlHKF5RM6aM7oj-Y4lv1CnQbGQcpatrbfc
CLIENT_INFO_ID=1CG07N6tYpIoPD_vp0cQ8lu_uMAVO4NRwuvL_J6-fTe8
ORDER_RECORD_ID=1Qi7IuVjksPQa3wv_YIid_UCHaHYmmmyBH3oT8BJKLIk
QBO_MODE=production    ← 本番モードに切替
ALERT_EMAIL=ordercfp@gmail.com
... (その他QBO認証情報等)
```

#### Step 6：本番デプロイ完了確認

- [ ] Render のログでエラーが出ていないか確認
- [ ] `https://order-system-5zq7.onrender.com` にアクセスして起動確認

### 段階3の完了条件

移植6ステップがすべて完了したら段階4へ。

## 5. 段階4：本番環境で自社（MY Inc.）テスト注文

### 目的
本番環境で実際にQBOへインボイスが作成されるが、**自社宛なので業務影響なし**で確認できます。

### 5-1. 事前準備

1. QBO に **「California Food Product (Test)」** などの自社顧客レコードがあるか確認
   - 無ければ作成
2. 本番 Client list にも同じ顧客を追加（または既存のテスト顧客を使う）
3. Customer ID を割り当て（例：000 or 999）

### 5-2. テスト手順

#### テストA：Standardパターン

1. テスト顧客の Price Group を **Standard** に設定
2. QRコード注文フォームまたは API でテスト注文（5アイテム程度）
3. QBO にインボイスが作成されるか確認
4. 単価が **Item List J列 or K列の値そのまま**であることを確認

- [ ] QBO インボイスが作成される
- [ ] 全アイテムで Standard 価格が反映
- [ ] 量り売り商品の単価が J列
- [ ] 定量売り商品の単価が K列
- [ ] PrivateNote に「standard」と記録

#### テストB：Group Aパターン

1. テスト顧客の Price Group を **Group A** に変更
2. 同じテスト注文
3. QBO のインボイスを確認

- [ ] 単価が Item List の値 × 1.015 になっている
- [ ] 小数点以下2桁で丸められている
- [ ] PrivateNote に「group-a」と記録

#### テストC：Individualパターン（Custom Price あり）

1. テスト顧客の Price Group を **Individual** に変更
2. Custom Prices に1〜2件のテスト価格を追加
3. 同じテスト注文
4. QBO のインボイスを確認

- [ ] Custom Price 登録済みSKU → Custom Price が反映
- [ ] Custom Price 未登録SKU → Standardにフォールバック
- [ ] PrivateNote に各SKUの source が記録

### 5-3. テスト後のクリーンアップ

- [ ] QBO のテストインボイスを削除（または無視）
- [ ] テスト顧客の Price Group を最終的な設定に戻す
- [ ] テスト用 Custom Prices を削除

### 段階4の完了条件

3パターンすべてが期待通り動いたら段階5へ。

## 6. 段階5：先行顧客1〜2社で実運用テスト

### 目的
実際の顧客の注文で、本番運用と同じ条件で正常動作するかを確認します。

### 6-1. 先行顧客の選定基準

以下を満たす1〜2社を選定：

- [ ] **関係が良好**で、何かあれば連絡してくれる
- [ ] **注文頻度が比較的低い**（週1〜2回程度）
- [ ] **Individual分類で、Custom Pricesがシンプル**（2〜3アイテム程度）
- [ ] **Manabuさん自身が直接対応している顧客**

### 6-2. 実施手順

```
1. 先行顧客にメールで連絡（任意）
   「システム改修を行いました。引き続き同じ価格で配送されますが、
    もし請求書に違和感があればすぐにご連絡ください」
2. 通常通り注文を受ける（先行顧客 + その他は従来通りChoco経由でもOK）
3. インボイス送信後、Manabuさんが手動で価格を確認：
   - 過去のインボイスと同じ価格になっているか
   - PrivateNote に source が記録されているか
4. 1週間〜2週間モニタリング
5. 問題なければ次へ
```

### 6-3. モニタリング項目

毎日 or 注文ごとに確認：

- [ ] アラートメールが届いていないか（`ordercfp@gmail.com`）
- [ ] order-system record に「失敗」記録がないか
- [ ] QBO インボイスの価格が想定通りか
- [ ] 顧客から請求金額についての問い合わせがないか

### 段階5の完了条件

1〜2週間モニタリングして、トラブルがなければ段階6へ。

## 8. 段階7：Group A・Standard顧客も含めた全顧客展開

2. **Standard顧客**も自作システム経由に切り替え

### 8-2. モニタリング項目（2週間）

- [ ] Group A顧客で +1.5% が正しく反映されているか
- [ ] Standard顧客で標準価格が反映されているか
- [ ] エラー率が許容範囲（1日1件以下）
- [ ] 顧客クレームがないか

### 段階7の完了条件

## 9. 段階8：Choco解約

### 目的
プロジェクトの究極目的である Choco 解約を実施します。

### 9-1. 解約前の最終チェック

- [ ] 過去2週間でアラートメール頻発していない
- [ ] 全顧客から請求金額のクレームなし
- [ ] QBO インボイス作成の失敗が月1件以下
- [ ] **自作システムを完全に信頼できる状態**

### 9-2. 解約手順

1. Choco のサポートに解約依頼
2. 解約日まで並行稼働
3. 解約完了

### 9-3. Choco解約後の運用

- 自作システムが唯一の注文経路に
- 価格管理は Custom Prices シート + 価格管理メニュー
- 月次でモニタリング・改善

## 10. トラブル発生時の対応フロー

### 10-1. アラートメール受信時

```
1. メール内容を確認
2. order-system record の該当行を確認
3. 影響範囲を判定（1件のみ or 全件 or 特定パターン）
4. 緊急性が高ければ：
   - 該当注文を手動でQBOに修正
   - 一時的に該当顧客の注文を従来方法（Choco等）に切り替え
5. 根本原因を調査
6. 必要なら設計・実装を修正（開発環境でテスト後に再デプロイ）
```

### 10-2. 顧客クレーム受信時

```
1. 該当注文のインボイスを確認
2. QBO の PrivateNote で source を確認
   - source=custom → Custom Pricesシートを確認
   - source=group-a → ベース価格を確認
   - source=standard → Item List J/K列を確認
   - source=fallback → 警告通り、Individualだが未登録
3. 正しい価格を確認
4. 必要なら手動修正
5. システムの設定を見直し
```

### 10-3. ロールバック手順

最悪の場合の本番ロールバック：

```
1. Renderで前バージョンをデプロイ（ロールバック）
2. または .env の QBO_MODE を一時的に変更
3. 顧客には個別連絡
4. 原因究明と再デプロイ
```

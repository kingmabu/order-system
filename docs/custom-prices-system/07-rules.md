# 07. 重要ルール・コーディング規約・禁止事項

このファイルは、本プロジェクト全体を通じて Claude Code が**常に守るべきルール**を定義します。

**実装中、迷ったら必ずこのファイルに戻ってください。**

---

## 1. プロジェクト全体の最重要原則

### 原則1：本番環境を守る

- 本プロジェクトは Manabu さんの**毎日の業務基盤**
- 開発中に本番環境を壊すと、**1日数百件の注文処理**が止まる
- どんな改修も「**本番が動き続けること**」が最優先

### 原則2：既存スクリプトを尊重する

- 既存スクリプトは長年積み上げてきた業務ロジックの結晶
- 「もっとキレイに書ける」と思っても、**勝手にリファクタリングしない**
- 改修は「追加のみ」「既存部分の変更禁止」が基本

### 原則3：段階的に進める

- 1ステップずつ進める
- ステップ完了を Manabu さんが確認 → 次へ
- 焦って複数ステップをまとめて進めない

### 原則4：不明点は必ず確認

- 「たぶんこうだろう」で進めない
- 不明点は**選択肢形式で Manabu さんに質問**
- 答えを得てから次に進む

---

## 2. 作業ルール

### 2-1. 手順は1ステップずつ

- 大きなタスクを複数の小ステップに分解
- **1ステップ完了 → 動作確認 → Manabu さんの OK → 次へ**
- ステップごとに必ず動作確認

### 2-2. あいまいな指示は選択肢で確認

NG例：
```
「どうしますか？」
```

OK例：
```
「以下のうちどれで進めますか？
 A. ○○方式（メリット：xxx、デメリット：yyy）
 B. △△方式（メリット：xxx、デメリット：yyy）
 C. その他相談したい」
```

### 2-3. 既存コード変更前の確認

**既存スクリプトを変更する前**には、必ず：

1. 現状コードを Manabu さんから提供してもらう（既に把握済みでも再確認）
2. 変更箇所を**明確に説明**してから変更
3. 削除や上書きは特に慎重に

### 2-4. コード提供のルール

- **完全版で提供**（部分的な差分だけにしない）
- 変更箇所には `// ← 変更` または `// ← 追加` コメントを付ける
- ファイル全体を提供して、コピペで置き換え可能にする

### 2-5. /clear タイミングの通知

以下のタイミングで、Manabu さんに「**/clearを打つタイミングです**」と伝える：

- **機能完成時**：1つの機能が完成して動作確認OKになったとき
- **エラー3回ループ時**：同じエラーで3回直しても解決しないとき
- **コンテキストが肥大化したとき**：会話が長くなりすぎたとき

これによりトークン効率を保ち、新しい思考で問題に取り組めるようにする。

## 3. コーディング規約

### 3-1. 言語ポリシー

| 要素 | 言語 |
|---|---|
| **変数名・関数名** | 英語（snake_case or camelCase）|
| **クラス名** | 英語（PascalCase）|
| **定数** | 英語（UPPER_SNAKE_CASE）|
| **コメント** | 基本日本語、技術用語は英語そのまま |
| **ログメッセージ** | 日本語OK、英語OK（統一）|
| **ユーザー向けメッセージ（UI、メール、ダイアログ）** | **日本語** |

### 3-2. Apps Script コーディング規約

#### ファイル先頭にヘッダーコメント
```javascript
/**
 * ファイル名.gs - 簡単な説明
 *
 * 役割：
 * - xxx
 * - yyy
 *
 * 配置先: Cost list スプレッドシート
 */
```

#### 関数のJSDoc
```javascript
/**
 * 関数の説明
 * @param {string} param1 パラメータ1の説明
 * @return {Object} 戻り値の説明
 */
function myFunction(param1) {
  // ...
}
```

#### 内部関数（プライベート）には末尾アンダースコア
```javascript
function _internalHelper_(arg) {
  // 外部に公開しない関数
}
```

#### 定数定義は冒頭にまとめる
```javascript
// ===== 列番号（1始まり） =====
const COL_SKU = 1;
const COL_PRICE = 5;

// ===== シート名 =====
const CP_SHEET = 'Custom Prices';
```

### 3-3. Node.js コーディング規約

#### モジュール構造
```javascript
// 1. 外部ライブラリのrequire
const axios = require('axios');
const { google } = require('googleapis');

// 2. 内部モジュールのrequire
const { loadAllItems } = require('./routes/sheets-client');

// 3. 定数
const CONST_VALUE = 100;

// 4. 関数定義

// 5. module.exports（最後）
module.exports = { /* ... */ };
```

#### async/await を優先
```javascript
// 推奨
async function fetchData() {
  const data = await fetch(url);
  return data.json();
}

// 非推奨（callbackの連鎖）
function fetchData(callback) {
  // ...
}
```

#### エラーハンドリングは try-catch
```javascript
try {
  const result = await someAsyncFunction();
  return result;
} catch (err) {
  console.error('[functionName]', err);
  await sendAlertEmail('エラー', err.message);
  throw err; // or 適切なフォールバック
}
```

### 3-4. 命名規則の具体例

| 用途 | 良い例 | 悪い例 |
|---|---|---|
| 関数（動詞）| `getCustomPrice`, `addInvoice` | `customPriceFunc`, `process` |
| 真偽値 | `isUnit`, `hasError` | `unit`, `error` |
| 配列（複数形）| `customers`, `prices` | `customerList`, `priceArray` |
| 定数 | `MAX_RETRY`, `DEFAULT_PRICE` | `maxRetry`, `default_price` |

## 4. 禁止事項

### 4-1. 絶対にやってはいけないこと

#### ❌ 本番スプレッドシートを開発中に触る
- 開発中は **`[DEV]` 付き名前のコピーを使う**
- 本番データを誤って削除・上書きする可能性を排除

#### ❌ 既存スクリプトのロジック変更
- 既存関数の中身を**書き換えない**
- 必要なら新規関数を作って、既存関数は触らない

#### ❌ シート削除・列削除
- 既存シートを削除しない
- 既存列を削除しない
- 万一必要なら、**事前に Manabu さんに必ず確認**

#### ❌ 既存トリガーの削除・変更
- onEdit、5分監視、毎週月曜などの既存トリガーは触らない
- 新規トリガーを追加するのはOK

#### ❌ QBO の本番データを開発中に変更
- 必ず `QBO_MODE=dry-run` で開発
- 本番モードで動かすのは段階4以降

#### ❌ 環境変数ファイルを Git にコミット
- `.env.development` も `.env.production` も `.gitignore` に必ず追加
- 認証情報の漏洩防止

#### ❌ Customer ID をハードコード
- 顧客識別は必ず Client list 経由
- 「Customer ID 011 だから〜」のような分岐をコードに書かない

---

### 4-2. 強く非推奨

#### ⚠ 過度な抽象化
- 「将来こう使うかも」で複雑な抽象クラスを作らない
- **今必要な機能**だけシンプルに実装

#### ⚠ ライブラリの追加
- 必要最小限のライブラリだけ追加
- 既に `package.json` にあるものを優先利用
- 新規追加が必要な場合は **必ず Manabu さんに確認**

#### ⚠ console.log の乱用
- デバッグ用 `console.log` は本番に残さない
- 必要な場合は `console.info` / `console.warn` / `console.error` を使い分け

## 5. データハンドリングのルール

### 5-1. Customer ID は必ず正規化

```javascript
// 入力されたIDは何形式でも正規化
const normalized = normalizeId(rawId); // → "011"
```

### 5-2. 数値は必ず Number() で変換

```javascript
// Sheets から取得した値は文字列の可能性あり
const price = Number(row[4]);
if (isNaN(price)) {
  // エラー処理
}
```

### 5-3. 文字列は必ず trim() して比較

```javascript
const sku = String(rawSku).trim();
const match = items.find(i => i.sku.trim() === sku);
```

### 5-4. 価格は小数点以下2桁で丸める

```javascript
function roundPrice(price) {
  return Math.round(price * 100) / 100;
}
```

### 5-5. 日付フォーマット

- **シート表示用**：`M/d/yyyy`（例：5/13/2026）
- **ログ用**：`yyyy/MM/dd HH:mm:ss`
- **API用**：ISO 8601（`new Date().toISOString()`）

## 6. エラーハンドリングのルール

### 6-1. 業務継続性優先

- 些細なエラーで全体を止めない
- 1アイテムでエラーが出ても、他のアイテムは処理を続行
- ただし、**通知は必ず行う**

### 6-2. リトライ可否の判断

| エラー種別 | リトライ |
|---|---|
| ネットワークタイムアウト | ✅ する（3回まで）|
| HTTP 5xx 系 | ✅ する（3回まで）|
| HTTP 4xx 系（401/403/404 等）| ❌ しない（即エラー）|
| データ不整合（NaN等）| ❌ しない（即フォールバック or スキップ）|

### 6-3. 指数バックオフ

```javascript
const waitMs = Math.pow(3, attempt - 1) * 1000; // 1秒, 3秒, 9秒
```

### 6-4. アラートメールの送信基準

| 状況 | 通知 |
|---|---|
| フォールバック発動（Custom Price未登録）| ✅ 警告メール |
| Customer ID 未マッチ | ✅ 警告メール |
| SKU 未マッチ | ✅ エラーメール |
| QBO API 失敗（全リトライ後）| ✅ エラーメール（緊急）|
| Sheets API 失敗（全リトライ後）| ✅ エラーメール（緊急）|

### 6-5. ログの出力レベル

```javascript
console.log(...)   // 通常の情報
console.info(...)  // 重要な情報（重複可）
console.warn(...)  // 警告（要監視）
console.error(...) // エラー（要対応）
```

## 7. テスト・デバッグのルール

### 7-1. 段階的に確認

- 1つの機能を実装 → すぐ動作確認 → OK なら次へ
- 複数機能を同時に実装しない

### 7-2. dry-run モードで開発

- 開発中は **必ず `QBO_MODE=dry-run`**
- コンソールに `[DRY-RUN]` ログが出ることを確認

### 7-3. ローカルテストで完結

- 本番にデプロイする前に、必ずローカルで動作確認
- `node server.js` を起動して curl でテスト

### 7-4. テストデータの扱い

- 開発中に登録したテストデータは、**本番移行前にクリアする**
- Custom Prices シートを開発用と本番用で**混同しない**

## 8. ドキュメントのルール

### 8-1. コード内コメント

- なぜそうしているかの**理由**を書く（何をしているかはコードを読めば分かる）
- TODO や FIXME には**理由と日付**を付ける

```javascript
// TODO(2026-06): 将来 Group B（+3%）対応を検討
// FIXME(2026-05-13): QBOのPriceLevel構造が組織により異なるため、本番投入前に再確認
```

### 8-2. 変更履歴の管理

- Gitコミットメッセージは具体的に
- 例：`feat: add Custom Prices form` / `fix: handle empty SKU in Cost Reference`

### 8-3. README の更新

- 新機能を追加したら README にも反映
- 既存の使い方を変えた場合は**特に明記**

## 10. セキュリティのルール

### 10-1. 認証情報の管理

- 認証情報は**環境変数のみ**
- コード内にハードコードしない
- Gitにコミットしない

### 10-2. ログに認証情報を出さない

```javascript
// NG
console.log('Token:', accessToken);

// OK
console.log('Token: [REDACTED]');
```

### 10-3. エラーメッセージの注意

- 顧客名・価格などの**業務データ**はアラートメールに含めてOK
- ただし、**認証情報・APIキー**は絶対に含めない

## 11. パフォーマンスのルール

### 11-1. 一括取得を優先

- ループ内で API を叩かない
- 注文処理の最初に**1回だけ**一括取得

### 11-2. Sheets API の節約

- `getValues()` は範囲を絞って実行
- 不要な列まで取得しない

### 11-3. 適切なタイムアウト

- 各 API 呼び出しに 30秒程度のタイムアウトを設定
- 無限に待たないようにする

## 12. メンテナンスのルール

### 12-1. 既存トリガーの確認

開発中は既存の以下のトリガーを**意識**：

- `onEdit`（Cost list）
- 5分監視 `watchVendorPrices`（Cost list）
- 毎週月曜13:00 `createWeeklyDraftFromCostList`（Cost list）

→ 新規シート3つで誤動作しないことを確認

### 12-2. Discontinuedタブとの兼ね合い

- 既存のDiscontinuedタブ作成機能は維持
- 新規シートはDiscontinuedとは別物として扱う

### 12-3. CFP Operations との連携

- 本プロジェクトは CFP Operations と直接的な連携はないが、将来連携する可能性あり
- そのときに困らない設計（モジュール分離）を心がける
